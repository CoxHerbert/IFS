# 智能出货工具需求文档

## 1. 产品定位

本系统面向客户和内部业务人员，核心不是单纯的门户展示，而是一个偏工具类的智能出货工作台。目标是减少客户整理货物数据、判断出货方式、计算成本、生成计划和跟踪状态的时间。

客户可以导入或录入货物明细，系统自动标准化数据，分析适合走散货还是整柜，生成成本测算、柜型建议、装柜模拟和出货分析报告。客户确认后，可以根据报告生成出货计划。内部后台按客户负责业务员做数据归属，业务员只能维护自己负责客户的出货计划状态。

## 2. 角色分类

### 2.1 客户用户

客户用户使用客户端/客户中心，主要能力包括：

- 录入或导入货物明细。
- 查看系统生成的出货分析报告。
- 下载分析报告。
- 根据报告生成出货计划。
- 查看出货计划列表。
- 进入出货计划详情页查看完整信息。
- 通过移动端或 PC 查看分享页。

### 2.2 后台用户/业务员

业务员指后台系统用户，即 `sys_user` 中的用户。每个客户可以指定一个固定负责业务员。

业务员主要能力包括：

- 查看自己负责客户的资料。
- 查看自己负责客户的出货计划。
- 更新对应客户的出货计划状态。
- 维护计划过程中的异常、费用、付款等信息。

### 2.3 管理员/全数据权限角色

管理员或拥有全数据权限的角色可以：

- 查看全部客户。
- 查看全部出货计划。
- 调整客户负责业务员。
- 管理角色、权限、客户账号和基础配置。

## 3. 客户端智能出货助手

### 3.1 页面流程

客户端智能出货助手采用步骤条布局：

1. 第一步：录入货物明细。
2. 第二步：生成出货分析。
3. 第三步：可根据报告生成出货计划。

### 3.2 货物明细录入

支持两种录入方式：

- 手动录入货物明细。
- 导入 Excel。

Excel 格式未知，用户不一定知道应该按什么模板导入，因此系统需要支持灵活识别字段。

字段识别要求：

- 识别货名、SKU/货号/唛头。
- 识别箱数、件数。
- 识别长宽高，支持独立列，也支持类似 `60x40x50` 的尺寸格式。
- 识别重量字段。
- 识别体积字段。
- 尽量兼容中文、英文和常见别名。

### 3.3 体积和重量语义

货物明细中的体积和重量应明确语义，避免歧义：

- `总重量(KG)`：该行货物的总重量，不是单箱重量。
- `总体积(CBM)`：该行货物的总体积，不是单箱体积。
- `单箱长/宽/高(cm)`：单个箱子的尺寸。
- 如果用户填写了单箱尺寸和箱数，但未填写总体积，系统自动计算该行总体积。
- 如果用户填写总体积，以总体积为准。

后续可以扩展支持：

- 单箱重量。
- 单箱体积。
- 用户导入时选择字段语义。

## 4. 出货分析计算

### 4.1 散货分析

散货需要计算：

- 总 CBM。
- 总重量。
- 散货单价，例如每 CBM 多少钱。
- 最低收费。
- 附加费用。
- 散货总成本。

散货适合场景：

- 体积较小。
- 走整柜成本明显更高。
- 装柜利用率较低。

### 4.2 整柜分析

整柜不能只按理论 CBM 判断。因为柜内尺寸和货物尺寸之间会有空隙，计算出 28CBM 并不代表一定能装进 20GP。

整柜计算需要考虑：

- 柜型内部长宽高。
- 柜型安全装载体积。
- 柜型最大重量。
- 货物单箱长宽高。
- 是否可堆叠。
- 易碎、不可堆叠、托盘、设备类货物的装载效率折损。
- 实际装柜风险等级。

系统应输出：

- 推荐柜型，例如 `20GP`、`40GP`、`40HQ`。
- 每种柜型需要多少个柜。
- 柜型单价。
- 附加费用。
- 整柜总成本。
- 装载率。
- 风险提示。

### 4.3 出货方式推荐

系统根据散货和整柜结果给出推荐：

- 推荐散货。
- 推荐整柜。
- 推荐具体柜型和柜量。
- 推荐原因。
- 成本差异。
- 风险等级。
- 推荐置信度。

## 5. 装柜模拟视图

当系统推荐整柜方案时，需要根据货物信息生成模拟装柜视图。

模拟视图需要展示：

- 柜型。
- 柜内长宽高。
- 货物摆放块。
- 不同货物使用不同颜色。
- 每个货物块展示货名、SKU、数量。
- 装载率。
- 超限或无法放入的风险提示。

注意：装柜模拟属于辅助判断，不等同于正式装柜方案。正式装柜仍需要结合仓库实际操作、货物包装、重心、托盘和装卸限制确认。

## 6. 出货分析报告

### 6.1 报告内容

分析报告应包含：

- 货物汇总。
- 标准化后的货物明细。
- 散货测算。
- 整柜测算。
- 系统推荐方案。
- 成本对比。
- 装柜模拟结果。
- 风险提示。
- 备注说明。

### 6.2 报告下载

客户可以下载报告。初期可以支持 HTML 或 PDF，后续可扩展 Excel、PDF 模板或带公司品牌样式的正式报告。

### 6.3 根据报告生成出货计划

客户可以选择根据报告生成出货计划。

生成计划时需要填写或确认：

- 客户订单号/参考号。
- 起运港。
- 目的港。
- 计划 ETD。
- 计划 ETA。
- 备注。
- 系统推荐的柜型/散货方案。
- 标准化后的货物明细。

生成后，客户可以在出货计划列表中继续跟踪。

## 7. 出货计划

### 7.1 列表页

客户端点击出货计划时，列表页只展示列表数据，不展示复杂详情。

列表建议展示：

- 计划编号。
- 客户订单号。
- 起运港/目的港。
- 当前状态。
- 总箱数。
- 总体积。
- 总重量。
- 创建时间。
- 更新时间。

点击计划编号或详情按钮进入详情页。

### 7.2 详情页

出货计划详情页使用 Tab 布局，一个 Tab 对应一类信息。

建议 Tab：

- 概览。
- 状态进度。
- 柜型方案。
- 货物明细。
- 费用付款。
- 异常费用。

详情页需要展示：

- 计划基础信息。
- 创建时间。
- 更新时间。
- 当前状态。
- 状态流转记录。
- 柜型方案。
- 货物明细。
- 费用和付款情况。
- 查验、异常和额外费用。

### 7.3 作废能力

出货计划应支持作废能力。

建议设计：

- 增加作废状态，例如 `cancelled` 或状态码 `910`。
- 作废需要填写原因。
- 已作废计划不能继续更新普通物流状态。
- 作废记录保留，不能物理删除。

### 7.4 计划结束条件

所有出货计划只有在客户完成付款后才算真正结束。

建议结束判断：

- 物流状态已完成，例如已签收。
- 应收费用已确认。
- 客户已付款。
- 无未处理异常费用。

满足以上条件后，计划状态可进入已完成/已结案。

## 8. 费用和付款管理

后续需要在出货计划详情中支持费用管理。

费用类型包括：

- 基础运费。
- 散货费用。
- 整柜费用。
- 报关费用。
- 文件费用。
- 仓储费用。
- 查验费用。
- 异常费用。
- 其他附加费用。

付款字段包括：

- 应收金额。
- 已收金额。
- 未收金额。
- 是否付款。
- 付款时间。
- 付款凭证。
- 付款备注。

异常费用需要记录：

- 费用原因。
- 发生时间。
- 金额。
- 是否需要客户承担。
- 是否已确认。
- 是否已付款。

## 9. 分享页

分享功能需要是移动端优先视图，同时 PC 端也可以完整查看。

要求：

- 移动端展示完整信息，不能只展示摘要。
- PC 端保持可读性和完整性。
- 不要求登录即可通过分享链接查看。
- 展示出货计划概览。
- 展示状态进度。
- 展示柜型方案。
- 展示货物明细。
- 展示费用和付款摘要。
- 展示异常提示。

分享页需要避免只做简单列表，应适合客户在手机上转发和查看。

## 10. 客户端 Header

客户端需要支持全屏功能。

要求：

- 全屏入口放在客户端 Header 中。
- 支持进入全屏和退出全屏。
- 图标状态应随当前是否全屏变化。

## 11. Agent 模型选择

Agent 支持模型选择。

要求：

- 用户可以自行选择模型。
- 后期可能存在多个模型。
- 新建会话、发送消息、分析文件时使用用户当前选择的模型。
- 历史会话如果已有模型，应展示该会话使用的模型。

模型列表应由后端接口返回，前端不应写死不可扩展逻辑。

## 12. Agent 会话审计

当前 Agent 对话按用户隔离：

- 后台用户只能看到自己创建的后台 Agent 会话。
- 客户中心用户只能看到自己客户账号创建的 Agent 会话。
- 客户账号会话和后台用户会话需要隔离，避免普通后台用户直接看到客户隐私对话。

后续需要新增后台独立的 Agent 会话管理/会话审计功能，不建议混在当前“我的 Agent 对话”页面中。

管理员或拥有 Agent 审计权限的角色可以：

- 查看全部后台用户和客户账号的 Agent 会话。
- 查看指定会话的完整消息记录。
- 按用户、客户、业务员、模型、时间、关键词筛选。
- 区分会话来源，例如后台用户会话、客户中心会话。
- 查看消息中的结构化分析结果，例如出货分析、表单、动作结果。

普通业务员建议只能查看自己负责客户的客户 Agent 会话，不应查看其他业务员客户的会话。

会话审计页面建议字段：

- 会话标题。
- 会话来源。
- 所属用户/客户账号。
- 关联客户。
- 负责业务员。
- 使用模型。
- 最近消息时间。
- 消息数量。
- 创建时间。
- 更新时间。

接口层要求：

- 当前个人聊天接口继续按 `currentUserID` 隔离。
- 新增独立审计接口，例如 `/agent/chat/audit/sessions` 和 `/agent/chat/audit/session/:sessionId/messages`。
- 审计接口必须配置独立权限标识，例如 `ifs:agent:audit`。
- 管理员/全数据权限角色可看全部。
- 普通业务员访问审计接口时，需要按客户负责业务员做数据过滤。

## 13. 后台角色和数据权限

### 13.1 业务员归属

业务员就是后台系统用户，即 `sys_user`。

客户表记录：

- `sales_user_id`：负责业务员用户 ID。
- `sales_user_name`：负责业务员名称。

出货计划表记录：

- `sales_user_id`：计划创建或绑定客户时同步的负责业务员 ID。
- `sales_user_name`：计划创建或绑定客户时同步的负责业务员名称。

出货计划保存业务员快照，避免客户后续更换业务员时影响历史计划归属。

### 13.2 后台权限规则

管理员或全数据权限角色：

- 可以查看全部客户。
- 可以查看全部出货计划。
- 可以更新全部出货计划状态。
- 可以调整客户负责业务员。
- 可以在具备 Agent 审计权限时查看全部 Agent 会话。

普通业务员：

- 只能查看自己负责的客户。
- 只能查看自己负责客户的出货计划。
- 只能更新自己负责客户的出货计划状态。
- 不能维护其他业务员客户的出货计划。
- 如开放 Agent 审计，只能查看自己负责客户的客户 Agent 会话。

### 13.3 接口权限要求

以下接口需要做数据归属校验：

- 出货计划列表。
- 出货计划详情。
- 出货计划状态更新。
- 出货计划绑定客户。
- 出货计划确认。
- 出货计划分享。
- 出货计划删除/作废。
- 客户列表。
- Agent 会话审计列表。
- Agent 会话审计详情。

前端隐藏按钮不能作为唯一权限控制，后端必须校验。

## 16. 验收标准

- 用户可以在不知道 Excel 固定格式的情况下导入常见货物数据。
- 系统能明确区分总体积/总重量和单箱尺寸。
- 系统能给出散货和整柜成本测算。
- 整柜推荐不能只依赖理论 CBM，需要考虑柜内尺寸和装载风险。
- 整柜方案能生成模拟装柜视图。
- 用户可以下载分析报告。
- 用户可以根据报告生成出货计划。
- 出货计划列表和详情分离。
- 出货计划详情使用 Tab 分类展示。
- 出货计划有创建时间和更新时间。
- 分享页在手机和 PC 都能完整查看。
- 后台客户可绑定负责业务员。
- 普通业务员只能更新自己负责客户的出货计划状态。
- 管理员或全数据权限角色可以管理全部数据。
- 当前个人 Agent 对话页只展示当前用户自己的会话。
- 管理员可通过独立 Agent 会话审计入口查看全部会话。
- 普通业务员在 Agent 会话审计中只能查看自己负责客户的客户会话。
